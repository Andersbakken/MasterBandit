- [x] DECSTR (`CSI ! p`) — Soft terminal reset. Restores mode/attribute state via `resetToDefault` + preserves xterm extensions (mouse, focus, paste, sync, color-pref) and cursor position/shape/blink per VT510.
- [x] DECCOLM (private mode 3) — 80/132 column toggle. Window is user-controlled so no resize, but spec side effects (clear screen, home cursor, reset scroll region) apply on set and reset.
- [x] DEC line drawing / charset designation — `ESC ( X` / `ESC ) X` (ASCII, UK, DEC Special Graphics), SO (0x0E) / SI (0x0F) locking shifts, per-screen charset state saved/restored by DECSC/DECRC/SCP/RCP. Translation happens at the ASCII write site — cells store final Unicode codepoints (U+2500 series etc.), rendered by the existing procedural-glyph pipeline. SS2/SS3 parsed but no-op (G2/G3 slots not implemented).
- [x] DECOM (private mode 6) — Origin mode. CUP/HVP/VPA coordinates relative to scroll region when set; DSR cursor-position report adjusts to relative row. Save/restore in DECSC/DECRC. DECRQM `CSI ? 6 $ p` reports status.
- [ ] Mode 1047 / 1048 / 47 — alt-screen without save-cursor, save-cursor without alt-screen, and legacy alt-screen. Companions to the already-implemented 1049. Rarely emitted in isolation but trivial to add.
- [ ] OSC 52 destination selectors — the current handler accepts `c;...` and any non-`c` prefix silently maps to the same clipboard callback. Per spec, destinations are `c` (clipboard), `p` (primary), `s` (selection), `q`, and cut buffers `0`–`7`; they should route through distinct callbacks (e.g. `copyToPrimary` / `copyToSelection`) so Linux middle-click paste and SSH-forwarded primary selections work correctly.
- [ ] OSC 633 — VSCode shell integration. Alternative to OSC 133 with subcommands `A` (prompt start), `B` (command start), `C` (command executed), `D` (command finished + exit code), `E` (command line), `P` (property). Cheap to add — translate each subcommand to the existing OSC 133 semantic-tag plumbing and `CommandRecord` ring. Without it, VSCode's integrated terminal can't detect prompts/commands from our emulator.
- [ ] XTMODKEYS / XTQMODKEYS (`CSI > Pp m`, `CSI ? Pp m`) — Modifier-key encoding knob. Largely superseded by kitty keyboard protocol; some apps still probe it.
- [ ] DECSCA (`CSI Ps " q`) — Character protection attribute. Only consumed by very old VT apps; safe to ignore but worth recognising to silence `Unknown CSI` for final byte `q` with intermediate `"`.

## Architecture / Threading

Two-thread model: main thread owns the event loop, PTY reads, VT parsing, window/input, and script execution; a dedicated render thread owns GPU submission. Render reads from `TerminalSnapshot` (viewport copy) under `TerminalEmulator::mMutex` (`std::recursive_mutex`). Wakeups via `renderCv_` / `renderWake_`.

The coarse mutex, `PendingMutations`, and `RenderFrameState` shadow copy live on `RenderThread` (exposed only to `PlatformDawn` via `friend`). Main thread accumulates mutations in `renderThread_->pending()` (no lock needed — main-thread-only), then applies them in a single batch via `renderThread_->applyPendingMutations()` under `renderThread_->mutex()` at the end of each event loop tick. This rebuilds `renderThread_->renderState()` — a shadow copy of tab/pane structure, font metrics, colors, divider geometry, and pre-computed tab bar cells. The render thread (inside `RenderEngine::renderFrame`) snapshots `renderState()` into its own `frameState_` under the same mutex at frame start, then releases the lock for the remainder of the frame. All post-unlock reads go through `frameState_`; render-thread-only state lives in `RenderEngine`'s `PaneRenderPrivate` maps. Zero GPU/Dawn calls on the main thread after init. See `src/platform/RenderSync.h`, `src/platform/RenderThread.{h,cpp}`, `src/platform/RenderEngine.{h,cpp}`, `src/platform/PlatformDawn.{h,cpp}`, and `src/terminal/TerminalSnapshot.{h,cpp}`.

- [ ] Script callbacks during parsing — `TerminalCallbacks` fire from `TerminalEmulator::parseBatch()` on the main thread (title, icon, CWD, progress, OSC handlers, `commandComplete`, `foregroundProcessChanged`, etc.), running QuickJS synchronously under `TerminalEmulator::mMutex`. A `deferredMain_` / `postToMainThread()` drain already exists and some lifecycle callbacks route through it, but the parse-time OSC and command-lifecycle callbacks still run in-line. Two problems: (1) **stall** — a long-running or I/O-blocking script holds `mMutex` for its whole execution, which makes the render thread's `snapshot.update()` for that terminal block for the same duration and spikes frame latency. (2) **parser re-entry** — a script that calls `pane.inject()` mid-parse appends bytes into the parse state machine while it's mid-sequence, risking DCS/OSC accumulation and CSI parameter buffers getting interleaved with the script's own writes. (Not a deadlock: `mMutex` is recursive and the render thread never chains across multiple `mMutex` instances, so there's no circular wait available.) Route the remaining parse-time callbacks through the post-parse drain so `mMutex` is released and the parser is in a clean state before JS runs; consider a per-script wall-time budget for the drain to bound tail latency.
- [x] Config reload can invalidate render caches mid-frame — fixed: `PlatformDawn::applyConfig()` (invoked by `ConfigLoader`) holds `renderThread_->mutex()` for its duration so font/color/atlas mutations can't overlap with the render thread's shaping phase. Render thread reads config values from `frameState_` (snapshotted under lock).
- [ ] `RenderThread::mutex()` ownership assertion wrapper — wrap the mutex owned by `RenderThread` in a class that tracks the owning `std::thread::id` (set after lock, cleared before unlock). Add an `assertHeld()` method that asserts `owner == std::this_thread::get_id()`. Call it at the top of functions like `applyFramebufferResize()`, `buildRenderFrameState()`, `TabManager::refreshDividers()` that expect the caller to already hold the lock. Debug-only (no-op in release builds).
- [ ] Merge `pendingExits_` / `deferredExitMutex_` into `deferredMain_` — push `[this, t]{ terminalExited(t); }` instead of the raw `Terminal*`. Eliminates one mutex.
- [ ] Audit redundant event loop wakeups — Instruments shows ~17ms/sample of `mach_msg2_trap` from `CFRunLoopWakeUp` on the main thread during animation playback. Each `eventLoop_->wakeup()` posts a Mach message; several call sites are redundant: (1) the end-of-tick `wakeup()` in `onTick` when `needsRedraw_` or `pendingGpuCallbacks_` is set — the run loop is already running, it will iterate naturally; (2) `postToMainThread()` when already on the main thread — should run the lambda directly; (3) `setNeedsRedraw()` calling `wakeup()` when the render thread (not the event loop) is the consumer of `needsRedraw_`; (4) animation timer reschedules when a wakeup is imminent anyway. Audit and drop redundant calls.
- [ ] Demote `RenderThread::mutex_` from `std::recursive_mutex` back to `std::mutex`. It was made recursive because JS-initiated structural mutations (e.g. `scbs.destroyPopup` when a popup calls `popup.close()` from its own input handler) re-enter through the `InputController::onKey` → `Terminal::keyPressEvent` → popup input callback → JS → destroy path while the input handler is still holding the lock. Either (a) stop holding `platformMutex` across `Terminal` callbacks in `InputController::onKey`/`onChar`/`onMouseButton` (acquire only around tab/pane structural reads, release before the keyPressEvent/text/mouse dispatch), or (b) route JS-initiated structural destruction through `postToMainThread()` so the destroy runs on a fresh stack after the callback has unwound. Option (a) is the "pure" fix; option (b) adds one tick of latency between `popup.close()` and physical teardown. Until then the recursive mutex is load-bearing and must stay.
